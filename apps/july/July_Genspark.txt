Sistema de memoria continua para agentes IA — contexto entre sesiones, proyectos y herramientas

Fecha: 2026-03-12
Autores: Pendiente de asignación por equipo

📌 VISIÓN DEL PROYECTO
Construir un ecosistema personal de IA que nunca olvide: un sistema donde cada conversación, cada solución técnica, cada nota o idea quede almacenada, indexada y disponible para cualquier agente (Claude, Codex, VS Code Copilot, Z.ai...) en cualquier proyecto futuro.

Problema que resuelve:

Los LLMs olvidan todo al cerrar sesión
Cuando el contexto se comprime, se pierde información crítica
El conocimiento acumulado en un proyecto no viaja al siguiente
Las notas, vídeos de YouTube e ideas personales no están accesibles para el agente
Resultado esperado: Un agente con memoria persistente y búsqueda semántica que, al iniciar cualquier sesión, sepa quién eres, qué has hecho antes y cómo lo resolviste.

📝 FASE 2 — MEMORY PROTOCOL: ENSEÑAR AL AGENTE CUÁNDO RECORDAR
Objetivo: Que el agente sepa automáticamente cuándo guardar y cuándo buscar
Duración estimada: 1 hora
Hito: El agente termina cada sesión con un resumen guardado

El Memory Protocol es el prompt de sistema que enseña al agente el comportamiento correcto. Sin él, el agente tiene las herramientas pero no sabe usarlas.

2.1 — Dónde añadir el Memory Protocol
Agente	Archivo de configuración
Claude Code	~/.claude/CLAUDE.md
Cursor	.cursor/rules/memory.mdc (global o por proyecto)
VS Code Copilot	.github/copilot-instructions.md
OpenCode	.opencode/AGENT.md
2.2 — Contenido del Memory Protocol (añadir al prompt de sistema)
Copy## MEMORY PROTOCOL — REGLAS OBLIGATORIAS

### CUÁNDO GUARDAR (mem_save) — SIEMPRE, no es opcional
Llama a `mem_save` INMEDIATAMENTE después de cualquiera de estos eventos:
- Bug resuelto
- Decisión de arquitectura tomada
- Descubrimiento no obvio sobre el código
- Cambio de configuración o setup de entorno
- Patrón establecido (naming, estructura, convención)
- Preferencia o restricción del usuario aprendida
- Integración completada (MCP, API, librería...)

**Formato obligatorio para mem_save:**
- title: Verbo + qué (ej: "Integrado MCP de Engram en Claude Code")
- type: bugfix | decision | architecture | discovery | pattern | config | preference
- scope: project (por defecto) | personal
- topic_key: clave estable para temas evolutivos (ej: "architecture/auth-model")
- content:
  **What**: Una frase — qué se hizo
  **Why**: Qué lo motivó
  **Where**: Archivos o rutas afectadas
  **Learned**: Gotchas, casos edge, sorpresas (omitir si no hay)

### CUÁNDO BUSCAR (mem_search / mem_context)
Busca en memoria cuando el usuario use palabras como:
"recuerda", "cómo lo hicimos", "qué hicimos", "acuérdate", "en el otro proyecto"...

1. Primero: `mem_context` — contexto reciente de sesiones anteriores (rápido)
2. Si no encuentra: `mem_search` con palabras clave relevantes
3. Si hay match: `mem_get_observation` para contenido completo sin truncar

También busca PROACTIVAMENTE al empezar algo que pueda haber sido resuelto antes.

### PROTOCOLO DE CIERRE DE SESIÓN — OBLIGATORIO
Antes de terminar una sesión o decir "listo", ejecuta `mem_session_summary` con:

## Goal
[En qué trabajamos esta sesión]

## Instructions
[Preferencias o restricciones del usuario — omitir si ninguna]

## Discoveries
- [Hallazgos técnicos, gotchas, learnings no obvios]

## Accomplished
- [Items completados con detalles clave]

## Next Steps
- [Qué queda por hacer en la próxima sesión]

## Relevant Files
- ruta/al/archivo — [qué hace o qué cambió]

ESTO NO ES OPCIONAL. Si lo omites, la siguiente sesión empieza ciega.

### DESPUÉS DE COMPRESIÓN DE CONTEXTO
Si ves un mensaje sobre compactación o reset de contexto:
1. PRIMERO: llama a `mem_session_summary` con el contenido compactado
2. LUEGO: llama a `mem_context` para recuperar contexto de sesiones previas
3. SOLO ENTONCES: continúa trabajando

🔄 FASE 5 — SINCRONIZACIÓN ENTRE DISPOSITIVOS / EQUIPO
Objetivo: Compartir memorias entre miembros del equipo y dispositivos
Duración estimada: 2–4 horas según opción elegida
Hito: Memoria de un miembro accesible para otro

Opción A — Git Sync (sin servidor, recomendado para empezar)
Configurar repo compartido:

Copy# Cada miembro del equipo, en su directorio de trabajo
cd ~/proyectos/equipo-cerebro

# Exportar nuevas memorias como chunk
engram sync

# Esto genera en .engram/chunks/ un archivo .jsonl.gz nuevo
# Hacer commit y push de ese chunk

git add .engram/
git commit -m "sync: memorias sesión [fecha]"
git push
Importar memorias de otros miembros:

Copygit pull
engram sync --import
Automatizar con un alias:

Copy# En ~/.zshrc o ~/.bashrc
alias brain-sync="engram sync && git add .engram/ && git commit -m 'sync: $(date)' && git push && git pull && engram sync --import"
Opción B — Engram Cloud (Postgres compartido, para equipo avanzado)
Setup del servidor (una sola vez, en servidor del equipo):

Copy# docker-compose.yml mínimo
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: engram_cloud
      POSTGRES_USER: engram
      POSTGRES_PASSWORD: CAMBIAR_PASSWORD_SEGURA
    ports:
      - "5433:5432"
    volumes:
      - engram_pg_data:/var/lib/postgresql/data

volumes:
  engram_pg_data:
Copydocker compose up -d

export ENGRAM_DATABASE_URL="postgres://engram:PASSWORD@localhost:5433/engram_cloud?sslmode=disable"
export ENGRAM_JWT_SECRET="secreto-aleatorio-minimo-32-caracteres-CAMBIAR"

engram cloud serve
Cada miembro del equipo:

⚠️ RIESGOS Y MITIGACIONES
Riesgo	Probabilidad	Impacto	Mitigación
Agente no ejecuta Memory Protocol	Alta	Alto	Forzarlo en CLAUDE.md / copilot-instructions.md
Base de datos SQLite se corrompe	Baja	Alto	engram export periódico como backup
Ruido en memorias (guardar demasiado)	Media	Medio	Solo guardar lo no obvio. Revisar TUI semanalmente
Conflictos en Git Sync	Baja	Medio	Chunks son inmutables, el manifest es append-only
Contexto de Obsidian demasiado grande	Media	Bajo	Usar búsqueda semántica, no inyectar todo el vault
Credenciales en cloud.json expuestas	Media	Alto	Permisos 0600 (Engram lo hace solo), nunca en git
