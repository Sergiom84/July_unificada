Quiero que me ayudes con lo siguiente:

Utilizo códex de OpenAI para programar y necesito ayudarle para que tenga más contexto y no tenga que repetir cada poco tiempo la información, por ello, accede y lee el artículo siguiente y dime, qué pasos seguir desde codex y mi ordenador para llevarlo a cabo 

https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f?utm_source=perplexity

Sí. El artículo de Karpathy propone crear una wiki persistente mantenida por el LLM, no solo subir documentos para que el modelo los “recupere” cada vez. La idea es tener tres capas: fuentes crudas inmutables, wiki en Markdown generada por el LLM, y un schema/instrucciones tipo AGENTS.md para decirle al agente cómo mantenerla.

Para Codex, esto encaja muy bien porque Codex lee archivos AGENTS.md antes de trabajar, tanto globales como por proyecto, y permite que esas instrucciones se acumulen por carpetas.

Paso 1 — Crea una carpeta local para tu “wiki de contexto”

En tu ordenador, crea una carpeta separada para la memoria de trabajo. Por ejemplo:

mkdir -p ~/codex-wiki/raw/assets
mkdir -p ~/codex-wiki/wiki/sources
mkdir -p ~/codex-wiki/wiki/entities
mkdir -p ~/codex-wiki/wiki/concepts
mkdir -p ~/codex-wiki/wiki/decisions
mkdir -p ~/codex-wiki/wiki/analyses

cd ~/codex-wiki
touch AGENTS.md README.md wiki/index.md wiki/log.md
git init

La estructura quedaría así:

codex-wiki/
  AGENTS.md
  README.md
  raw/
    assets/
  wiki/
    index.md
    log.md
    sources/
    entities/
    concepts/
    decisions/
    analyses/

raw/ será tu fuente de verdad: artículos, notas, PDFs, capturas, documentación, etc. El LLM puede leerlo, pero no debería modificarlo. wiki/ será la capa que Codex mantiene: resúmenes, páginas de conceptos, decisiones, comparativas y síntesis. Esto sigue la arquitectura del artículo: fuentes crudas, wiki generada y schema de instrucciones.

Paso 2 — Abre esa carpeta con Obsidian, opcional pero recomendable

Instala Obsidian y abre ~/codex-wiki como vault. No es obligatorio, pero te ayuda a navegar visualmente la wiki, ver enlaces, usar graph view y revisar lo que Codex va creando.

El artículo recomienda Obsidian como una especie de “IDE” para la wiki: tú navegas y revisas; el LLM escribe y mantiene los archivos.

También puedes instalar Obsidian Web Clipper para guardar artículos como Markdown. Obsidian indica que Web Clipper guarda contenido en Markdown dentro de tu vault, aunque las imágenes no se descargan automáticamente; para bajarlas puedes usar el comando Download attachments for current file.

Paso 3 — Crea el AGENTS.md principal

Abre ~/codex-wiki/AGENTS.md y pega esto:

# LLM Wiki Instructions for Codex

## Purpose

This repository is a persistent LLM-maintained knowledge base.

The user provides raw sources in `raw/`. Codex reads those sources and maintains the generated wiki in `wiki/`.

The goal is not only to summarize documents, but to incrementally build a structured, interlinked, self-correcting knowledge base.

## Directory structure

- `raw/`: immutable source material. Never edit or delete files here unless the user explicitly asks.
- `raw/assets/`: local images and attachments.
- `wiki/index.md`: content-oriented index of all wiki pages.
- `wiki/log.md`: chronological append-only log of ingests, queries, maintenance, and decisions.
- `wiki/sources/`: one page per source document.
- `wiki/entities/`: people, companies, products, projects, apps, libraries, APIs.
- `wiki/concepts/`: reusable concepts, patterns, ideas, technologies.
- `wiki/decisions/`: decisions, trade-offs, conclusions, architecture choices.
- `wiki/analyses/`: synthesized answers, comparisons, investigations, reports.

## Core rules

1. Treat `raw/` as read-only.
2. Treat `wiki/` as Codex-maintained.
3. Before answering, inspect `wiki/index.md` and any relevant pages.
4. When ingesting a new source, create or update all relevant wiki pages.
5. Use Obsidian-style wikilinks: `[[Page Name]]`.
6. Flag contradictions clearly instead of silently choosing one claim.
7. Add citations back to source files whenever possible.
8. Update `wiki/index.md` after every meaningful change.
9. Append an entry to `wiki/log.md` after every ingest, query, lint pass, or major update.
10. Prefer small, precise edits over rewriting large pages unnecessarily.

## Page conventions

Every generated wiki page should start with YAML frontmatter:

```yaml
---
type: source | entity | concept | decision | analysis | index | log
status: draft | active | needs-review | superseded
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources:
  - path/to/source.md
tags:
  - example
---
Ingest workflow

When the user says to ingest a source:

Read the source from raw/.
Create a source summary in wiki/sources/.
Extract important entities and update or create pages in wiki/entities/.
Extract important concepts and update or create pages in wiki/concepts/.
Identify decisions, recommendations, contradictions, open questions, and reusable insights.
Update wiki/index.md.
Append a dated entry to wiki/log.md.
Query workflow

When the user asks a question:

Read wiki/index.md.
Identify relevant pages.
Read those pages before answering.
If the answer produces a reusable insight, ask whether it should be saved, or save it directly if the user requested wiki maintenance.
When saving, create a page in wiki/analyses/ and update wiki/index.md and wiki/log.md.
Lint workflow

When the user asks to lint or maintain the wiki:

Check for:

orphan pages
missing backlinks
duplicated concepts
stale claims
contradictions
source pages not reflected in entity or concept pages
important concepts without their own page
pages missing YAML frontmatter
broken wikilinks

Produce a maintenance report and, if requested, apply fixes.

Style
Write in Spanish unless the source or user request requires English.
Be concise but precise.
Prefer tables for comparisons.
Use clear headings.
Do not invent source claims.
Mark uncertainty explicitly.

Esto es el equivalente práctico del “schema” que menciona Karpathy: el documento que le dice al agente cómo se estructura la wiki y qué flujos debe seguir. :contentReference[oaicite:5]{index=5}

## Paso 4 — Configura Codex para que cargue instrucciones persistentes

Si usas Codex CLI, instala o actualiza Codex:

```bash
npm i -g @openai/codex@latest

OpenAI documenta que Codex CLI se ejecuta localmente en tu terminal, puede leer/cambiar código y ejecutar comandos en el directorio seleccionado.

Ahora prueba que Codex lee tu AGENTS.md:

codex --cd ~/codex-wiki --ask-for-approval never "Resume las instrucciones activas que has cargado."

Codex debería mencionar reglas de AGENTS.md, la estructura raw/, wiki/, index.md, log.md, etc. OpenAI recomienda precisamente este tipo de verificación para confirmar que Codex carga las instrucciones activas.

Paso 5 — Añade tu primer documento fuente

Por ejemplo, guarda un artículo en:

~/codex-wiki/raw/2026-05-08-articulo-sobre-mi-app.md

Luego abre Codex en esa carpeta:

codex --cd ~/codex-wiki

Y dile:

Ingiere raw/2026-05-08-articulo-sobre-mi-app.md siguiendo el workflow de AGENTS.md.

Quiero que:
1. crees una página de fuente en wiki/sources/
2. actualices conceptos relevantes en wiki/concepts/
3. actualices entidades relevantes en wiki/entities/
4. añadas conclusiones reutilizables en wiki/decisions/ si procede
5. actualices wiki/index.md
6. añadas una entrada a wiki/log.md

Este paso replica la operación “Ingest” del artículo: el LLM lee una nueva fuente, crea resumen, actualiza páginas relacionadas, revisa conceptos y deja registro cronológico.

Paso 6 — Usa la wiki como memoria antes de programar

Cuando vayas a trabajar en una app concreta, haz una de estas dos opciones.

Opción A — Wiki separada y proyecto separado

Tienes:

~/codex-wiki/
~/mis-proyectos/mi-app/

Desde tu app puedes lanzar Codex así:

codex --cd ~/mis-proyectos/mi-app --add-dir ~/codex-wiki

El flag --add-dir permite dar acceso a directorios adicionales junto al workspace principal.

Entonces puedes pedirle:

Antes de modificar código, revisa ~/codex-wiki/wiki/index.md y las páginas relevantes de la wiki.

Usa ese contexto para implementar la siguiente mejora:
[describe aquí la mejora]

Esta opción es útil si quieres una memoria común para varias apps.

Opción B — Wiki dentro de cada proyecto

Dentro de tu proyecto:

mi-app/
  AGENTS.md
  src/
  docs/
  context/
    raw/
    wiki/

Esta opción es más simple para proyectos individuales. Codex encontrará el AGENTS.md del repositorio automáticamente cuando trabajes desde ahí, porque busca instrucciones desde la raíz del proyecto hasta el directorio actual.

Paso 7 — Crea un AGENTS.md también en cada app

En cada proyecto de programación, añade un AGENTS.md específico. Por ejemplo:

# Project Instructions

## Project context

This project uses the shared knowledge base at:

`~/codex-wiki/wiki/`

Before making architectural or product decisions, inspect the relevant wiki pages.

## Commands

- Install dependencies: `npm install`
- Start development server: `npm run dev`
- Run tests: `npm test`
- Run lint: `npm run lint`

## Coding rules

- Do not introduce new dependencies without explaining why.
- Keep changes small and reviewable.
- Update documentation when behavior changes.
- Prefer existing project patterns over new abstractions.

## Before coding

1. Read this file.
2. Read the project README.
3. Search the codebase for existing patterns.
4. If the task relates to prior decisions, read the shared wiki first.

## After coding

1. Run relevant tests.
2. Summarize changed files.
3. Explain trade-offs.
4. If a reusable project decision was made, add it to the shared wiki.

OpenAI recomienda usar AGENTS.md para dar a Codex instrucciones de navegación del código, comandos de prueba y prácticas estándar del proyecto.

Paso 8 — Usa tres comandos habituales con Codex
1. Ingerir nueva información
Ingiere este nuevo documento de raw/ y actualiza la wiki siguiendo AGENTS.md.
2. Preguntar usando la memoria
Antes de responder, lee wiki/index.md y las páginas relevantes. 
Con base en la wiki, dime qué decisión técnica conviene para [tema].
3. Hacer mantenimiento
Haz un lint de la wiki: busca contradicciones, páginas huérfanas, conceptos duplicados, claims obsoletos y enlaces rotos. 
Primero dame el informe. Después te diré qué cambios aplicar.

El artículo recomienda justamente tres operaciones recurrentes: ingest, query y lint.

Paso 9 — Añade búsqueda local cuando la wiki crezca

Al principio basta con wiki/index.md. Karpathy comenta que esto funciona sorprendentemente bien a escala moderada, alrededor de unas 100 fuentes y cientos de páginas.

Cuando crezca, puedes añadir una herramienta como qmd, que combina búsqueda BM25, búsqueda vectorial y reranking local para Markdown.

Pero no empieces por ahí. Primero crea la wiki simple con Markdown, Git, Obsidian y Codex.

Mi recomendación práctica

Empieza con esta versión mínima:

codex-wiki/
  AGENTS.md
  raw/
  wiki/
    index.md
    log.md
    sources/
    concepts/
    entities/
    decisions/
    analyses/

Y usa este flujo:

Guardas artículos, notas y decisiones en raw/.
Pides a Codex que los ingiera.
Codex actualiza wiki/.
Antes de programar, pides a Codex que revise wiki/index.md.
Cuando una conversación produzca una decisión útil, la guardas en wiki/decisions/.
Cada cierto tiempo pides un lint de la wiki.
